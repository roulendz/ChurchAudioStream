<objective>
Fix WSS admin WebSocket upgrade interference where protoo's WebSocket-Node unconditionally
consumes ALL upgrade requests on httpsServer, corrupting admin WSS connections.

Purpose: Admin connections via wss://<LAN_IP>:7777 currently connect and receive "welcome" but
"identify:ack" times out because WebSocket-Node (via protoo-server) and ws (admin WS) both
attempt to complete the WebSocket handshake on the same TCP socket simultaneously. The admin
ws handler runs first (sends welcome) but WebSocket-Node's concurrent handshake corrupts the
socket state. WS loopback on port 7778 works because only the admin handler is attached there.

Output: Single upgrade dispatcher pattern where handler.ts is the sole upgrade listener on
httpsServer, routing /ws/listener upgrades to protoo via a dummy HTTP server, and all other
paths to the admin ws server.
</objective>

  <action>
The root cause is that `new protooServer.WebSocketServer(httpsServer)` registers WebSocket-Node's
upgrade handler directly on httpsServer, competing with the admin ws handler. The fix:

1. In `listener-handler.ts`, change the constructor to NOT pass httpsServer to protoo. Instead:
   - Import `http` from `node:http`
   - Create a private dummy HTTP server: `this.dummyUpgradeServer = http.createServer()`
     (never listened on any port -- just an EventEmitter for upgrade events)
   - Pass `this.dummyUpgradeServer` to `new protooServer.WebSocketServer(this.dummyUpgradeServer)`
   - This satisfies WebSocket-Node's requirement for an httpServer while isolating its upgrade
     handler to the dummy server that will never receive real traffic on its own

2. Add a public `forwardUpgrade(request, socket, head)` method to `ListenerWebSocketHandler`:
   ```typescript
   forwardUpgrade(
     request: IncomingMessage,
     socket: Duplex,
     head: Buffer,
   ): void {
     this.dummyUpgradeServer.emit("upgrade", request, socket, head);
   }
   ```
   This manually triggers WebSocket-Node's upgrade handler on the dummy server when the
   dispatcher in handler.ts routes a /ws/listener upgrade to us.

3. Update the constructor signature: remove `httpsServer` parameter, replace with no server
   dependency. The constructor should accept `signalingHandler` and `config` only (no server).

4. In the `close()` method, also close the dummy server if needed (it's not listening so
   just null it out for GC).

5. Update `protoo-types.d.ts`: The WebSocketServer constructor type currently requires
   `HttpServer | HttpsServer`. This is correct -- we're now passing an `http.Server` (the
   dummy). No type change needed as `http.Server` already matches.

Important: Do NOT change the connectionrequest handler logic -- it already correctly validates
paths and handles rate limiting. The only change is WHERE protoo registers its upgrade handler
(dummy server instead of httpsServer).
