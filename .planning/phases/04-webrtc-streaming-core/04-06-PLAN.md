  <action>
    **Create `sidecar/src/streaming/latency-estimator.ts`:**

    LatencyEstimator provides component-based latency estimation per channel (per locked decision: no active measurement/test tones, just sum components).

    **`estimateLatency(channelConfig)` method:**
    Return LatencyEstimate with:
    - `gstreamerBufferMs`: Derived from Opus frame size (config.opus.frameSize in ms) + audioloudnorm buffer (~10ms for real-time mode)
    - `opusEncodeMs`: Equals frame size (encoding takes ~1 frame size)
    - `mediasoupProcessingMs`: Constant ~1ms (C++ worker forwards RTP, no transcoding)
    - `webrtcJitterBufferMs`: Depends on latency mode:
      - "live": ~20ms (minimal buffer)
      - "stable": ~60ms (default WebRTC jitter buffer)
    - `totalMs`: Sum of all components

    **`checkLatencyThreshold(estimate, thresholdMs = 200)` method:**
    - Returns boolean: true if totalMs exceeds threshold
    - Used to trigger admin warning (per locked decision: alert when estimated > 200ms)

    **Enhance `signaling-handler.ts` for Live/Stable mode:**

    When creating a Consumer, configure NACK/PLC based on channel's `lossRecovery` setting (per locked decision: per-channel admin setting):

    - **NACK mode (lossRecovery: "nack"):** Create consumer with default rtcpFeedback (mediasoup handles NACK retransmission in C++ worker). This is the default behavior -- just create the consumer normally.

    - **PLC mode (lossRecovery: "plc"):** Create consumer and then disable NACK by not including transport-cc and nack in the consumer's rtcpFeedback. Implementation: when creating consumer via `transport.consume()`, check if we need to override rtpParameters. If mediasoup doesn't support per-consumer NACK disable at creation time, document this as a limitation and note that the "Live" vs "Stable" mode primarily affects client-side jitter buffer configuration (which is Phase 5 listener UI territory).

    **Mapping per locked decision specifics:**
    - "Live" mode maps to: PLC preferred + minimal jitter buffer hint
    - "Stable" mode maps to: NACK enabled + default jitter buffer

    **NOTE:** The browser's jitter buffer configuration is actually controlled by mediasoup-client on the browser side (Phase 5). In Phase 4, the server-side distinction is:
    - Whether NACK retransmission is enabled for the consumer
    - The latency estimate calculation uses the mode to estimate jitter buffer delay
    - Channel metadata sent to listeners includes the mode so Phase 5 can configure accordingly
