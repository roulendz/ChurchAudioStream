    **`createForListener(router, peerId)` method:**
    - Create WebRtcTransport with:
      ```
      listenInfos: [{ protocol: "udp", ip: "0.0.0.0", announcedAddress: this.announcedIpAddress }]
      enableUdp: true
      enableTcp: false    // UDP-only for lowest latency on local WiFi (locked: no TURN server)
      preferUdp: true
      initialAvailableOutgoingBitrate: 128_000  // Audio-only: 128 kbps Opus
      iceConsentTimeout: 30
      ```
    - Register event handlers:
      - `transport.on("icestatechange")` -- log state changes; if "disconnected" or "closed", log for monitoring
      - `transport.on("dtlsstatechange")` -- if "failed" or "closed", close transport and clean up
      - `transport.on("routerclose")` -- clean up from tracking Map

    - Get transport from Map, call `transport.connect({ dtlsParameters })`

    **`getTransport(peerId)` method:** Returns transport from Map.

    **`closeTransport(peerId)` method:**
    - Close transport, remove from Map

    **`getTransportStats(peerId)` method:**
    - Returns transport stats for per-listener connection info

    **`closeAll()` method:**
    - Close all transports, clear Map

    **Create `sidecar/src/streaming/signaling-handler.ts`:**

    SignalingHandler class processes protoo peer requests and sends notifications.

    **Constructor:** Takes RouterManager, TransportManager, and a channel metadata resolver function.

    **`handlePeer(peer)` method:**
    - Initialize peer.data with ListenerPeerData: `{ sessionId: crypto.randomUUID(), connectedAt: Date.now(), rtpCapabilities: null, webRtcTransport: null, currentConsumer: null, currentChannelId: null, isAdmin: false }`
    - Push active channels immediately on connect via `peer.notify("activeChannels", { channels })` (per locked decision: server pushes channel list on initial connect)
    - Register request handler `peer.on("request", ...)` dispatching to:

      **`getRouterRtpCapabilities`:**
      - Return rtpCapabilities from any channel's router (all share same Opus codec). If no routers exist, return the codec capability directly.
      - Store `request.data.rtpCapabilities` (sent by client) into peer.data for later canConsume checks

      **`createWebRtcTransport`:**
      - Call `transportManager.createForListener(router, peer.id)`
      - Store transport reference in peer.data.webRtcTransport
      - Accept with transport info (id, iceParameters, iceCandidates, dtlsParameters)

      **`connectWebRtcTransport`:**
      - Extract `dtlsParameters` from request.data
      - Call `transportManager.connectTransport(peer.id, dtlsParameters)`
      - Accept()

      **`consume` (also handles initial channel subscription):**
      - Extract `channelId` from request.data
      - Get router and producer from RouterManager for channelId
      - If no producer: reject(404, "Channel not active") -- per locked decision: listeners connect but hear silence when no pipeline
      - Check `router.canConsume({ producerId, rtpCapabilities: peer.data.rtpCapabilities })`
      - If peer has currentConsumer, close it first (channel switch via consume)
      - Create consumer on peer's WebRtcTransport: `transport.consume({ producerId, rtpCapabilities, paused: true })` -- always paused (per research anti-pattern #4)
      - Wire consumer events:
        - `consumer.on("producerclose")` -- notify peer "consumerClosed", set currentConsumer to null
        - `consumer.on("transportclose")` -- set currentConsumer to null
      - Store consumer in peer.data.currentConsumer, update currentChannelId
      - Accept with `{ consumerId, producerId, kind, rtpParameters }`

      **`resumeConsumer`:**
      - Call `peer.data.currentConsumer.resume()`
      - Accept()

      **`switchChannel`:**
      - Extract `channelId` from request.data
      - Close current consumer if any
      - Create new consumer on target channel (same logic as consume)
      - If switch fails, attempt to restore previous channel; if that also fails, notify with remaining active channels (per locked decisions on channel switch failure)
      - Accept with new consumer info

    - Register close handler `peer.on("close", ...)`:
      - Close webRtcTransport via transportManager
      - Emit `"listener-disconnected"` with peer.data.sessionId

    **Heartbeat integration:**
    - Track last activity timestamp per peer
    - Implement `checkZombieConnections(heartbeatIntervalMs)` that pings peers and closes unresponsive ones (per locked decision: 30-60 second interval)

    **`getListenerCount(channelId?)` method:**
    - Count peers with currentChannelId matching (exclude isAdmin peers per locked decision)

    **`getListenerSessions()` method:**
    - Return ListenerSessionInfo[] for admin dashboard

    **`notifyAllListeners(method, data)` method:**
    - Send notification to all connected peers

    **`notifyListenersOnChannel(channelId, method, data)` method:**
    - Send notification to listeners on a specific channel

    **Create `sidecar/src/ws/listener-handler.ts`:**

    ListenerWebSocketHandler class manages the protoo WebSocketServer for listeners.

    **Constructor:** Takes HTTPS server, SignalingHandler, and streaming config (rateLimit, heartbeatInterval).

    **Setup:**
    - Create `protooServer.WebSocketServer(httpsServer, { ... })` -- protoo wraps ws internally
    - Handle `connectionrequest` event:
      - Parse `info.request.url` to validate path is `/ws/listener`
      - If path doesn't match, reject the connection
      - Apply rate limiting: track connection attempts per IP in a Map with sliding window
        - If IP exceeds `rateLimitPerIp` within `rateLimitWindowMs`, reject with 429
      - Accept the connection, get the protoo Transport
      - Create a new peer in the protoo Room: `room.createPeer(peerId, transport)`
      - Delegate to `signalingHandler.handlePeer(peer)`

    **Heartbeat:**
    - Start heartbeat interval (from config.streaming.heartbeatIntervalMs, default 30s)
    - Check all peers for zombie connections

    **`close()` method:**
    - Close all connections, close the protoo WebSocketServer

    **IMPORTANT path routing note:** protoo-server's WebSocketServer hooks into the HTTP server's "upgrade" event. Since the existing admin WebSocket (ws library) also uses "upgrade", we need to handle path-based routing in the HTTP server's upgrade event. The approach:
    - Remove the automatic upgrade handling from both ws and protoo
    - In server.ts, manually handle the "upgrade" event on httpsServer:
      - If `request.url` starts with `/ws/listener` -> route to protoo WebSocketServer
      - Otherwise -> route to existing ws WebSocketServer (admin)
    - This may require modifying server.ts to return the httpsServer reference so listener-handler.ts can attach to it
