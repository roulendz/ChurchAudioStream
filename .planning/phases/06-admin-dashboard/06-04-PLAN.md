  <name>Task 1: Create listener count and resource stats hooks</name>
  <files>
    src/hooks/useListenerCounts.ts
    src/hooks/useResourceStats.ts
  </files>
  <action>
    **Create `src/hooks/useListenerCounts.ts`:**
    - Accept `sendMessage` and `subscribe` functions.
    - State: `totalListeners: number` (useState), `channelCounts: Map<string, number>` (useState with Map).
    - On mount:
      - Send `streaming:status` to get initial snapshot.
      - Subscribe to `streaming:status` response: extract `totalListeners` and per-channel `listenerCount` from `channels` array.
      - Subscribe to `streaming:listener-count` broadcasts: update `channelCounts` map entry for `channelId` and `totalListeners`.
    - Expose: `{ totalListeners, getChannelListenerCount(channelId): number, channelCounts }`.
    - `getChannelListenerCount` returns `channelCounts.get(channelId) ?? 0`.
    - Cleanup: return unsubscribes.

    **Note:** useState is correct here (unlike audio levels). Listener count changes are infrequent (when listeners join/leave), not 10x/sec. React re-render on count change is desired to update badges.

    **Create `src/hooks/useResourceStats.ts`:**
    - Accept `sendMessage` and `subscribe` functions.
    - State (useState):
      ```typescript
      interface ResourceStats {
        uptime: number;          // seconds
        connections: { total: number; admin: number; listener: number; unidentified: number };
        config: { port: number; host: string; domain: string };
        cpuPercent: number | null;
        memoryMb: number | null;
      }
      ```
    - On mount:
      - Send `server:status` to get initial data.
      - Subscribe to `server:status` response: update stats from payload.
    - Set up a polling interval (every 10 seconds) to re-request `server:status` for uptime updates.
      (server:status is request-response, not push. CPU/memory are NOT pushed by server:status currently, but the uptime and connections ARE).
    - NOTE: The `stats:update` WS broadcast contains per-pipeline resource stats (from ResourceMonitor), NOT server-level CPU/memory. For server-level stats, we rely on the `server:status` request which gives uptime + connection counts. CPU and memory of the server process itself are not currently tracked. Set cpuPercent/memoryMb to null and display "N/A" -- they can be added in Phase 9.
    - Alternatively, check if `streaming:status` response includes worker info (it does: `workers` array with `peakMemoryKb`). Use that for mediasoup worker memory display.
    - Subscribe to `streaming:status` (sent by useListenerCounts already, or re-request): extract `workers` array for memory info.
    - Cleanup: clear polling interval, return unsubscribes.
    - Return `{ stats, workers }`.

    **Types:** Define local interfaces in each hook file.
  </action>
