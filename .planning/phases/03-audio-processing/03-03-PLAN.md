**channel-manager.ts** -- Major additions:

1. **Import new types:**
   - Import `ProcessingConfig, deriveSettingsFromMode, AudioModeType` from `../processing/processing-types.js`
   - Import `getPortsForChannel, generateSsrc` from `../processing/port-allocator.js`

2. **Debounce map:** Add a private field `private readonly restartDebounceTimers = new Map<string, NodeJS.Timeout>()` for per-channel debounce of processing config changes.

   Add constant: `const PROCESSING_DEBOUNCE_MS = 1500` (1.5 seconds -- within the 1-2s range specified by user).

3. **New method: `updateProcessingConfig(channelId: string, updates: Partial<ProcessingConfig>): AppChannel`**
   - Get channel or throw.
   - Deep-merge updates into `channel.processing`. For nested objects (agc, opus, rtpOutput), merge at the nested level:
     ```
     if (updates.agc) channel.processing.agc = { ...channel.processing.agc, ...updates.agc }
     if (updates.opus) channel.processing.opus = { ...channel.processing.opus, ...updates.opus }
     if (updates.rtpOutput) channel.processing.rtpOutput = { ...channel.processing.rtpOutput, ...updates.rtpOutput }
     if (updates.mode !== undefined) {
       channel.processing = deriveSettingsFromMode(updates.mode, channel.processing)
     }
     ```
   - Persist immediately: `this.persistChannels()`
   - If channel is streaming/starting, schedule debounced pipeline restart:
     ```
     this.scheduleDebouncedRestart(channelId)
     ```
   - Emit "channel-updated" event.
   - Return the updated channel.

4. **New method: `resetProcessingDefaults(channelId: string): AppChannel`**
   - Get channel or throw.
   - Get the channel's index in the channels array for port allocation.
   - Reset `channel.processing` to default ProcessingConfig (use defaults from processing-types.ts), preserving the channel's RTP port allocation.
   - Persist immediately.
   - If streaming, schedule debounced restart.
   - Emit "channel-updated".

5. **New private method: `scheduleDebouncedRestart(channelId: string): void`**
   - Clear any existing debounce timer for this channel.
   - Set a new timer: after PROCESSING_DEBOUNCE_MS, call `this.restartChannelPipelines(channelId)`.
   - Store timer in `restartDebounceTimers` map.

6. **New private method: `restartChannelPipelines(channelId: string): Promise<void>`**
   - Stop all pipelines for the channel (reuse `stopChannel` internals -- stop each pipeline, clear monitors).
   - Start all pipelines for the channel (reuse `startChannel` internals -- rebuild pipeline config, spawn new processes).
   - This effectively does a "restart" which produces new GStreamer processes with the updated processing config baked into the pipeline string.
   - Log the restart event.

7. **Modify `buildPipelineConfigFromAssignment`:**
   - After constructing the PipelineConfig (both AES67 and local variants), add the `processing` field from the channel's `channel.processing` config.
   - Compute port allocation: use `getPortsForChannel(channelIndex)` where `channelIndex` is derived from the channel's position in the persistent config array. Store the ports in `channel.processing.rtpOutput`.
   - Set SSRC: use `generateSsrc(channel.id)`.
   - Set `opus.audioType` based on `channel.processing.mode` (speech -> "voice", music -> "generic").
   - The result is a `PipelineConfig` with `.processing` set, which the pipeline builder (Plan 02) will use to construct the Phase 3 pipeline string.

8. **Modify `startPipelineForSource`:**
   - After starting the pipeline, call `this.levelMonitor.setProcessingTarget(pipelineId, channel.processing.agc.targetLufs)` so the level monitor can compute gain reduction.

9. **Modify `loadChannelsFromConfig`:**
   - When loading channels from config, populate `channel.processing` from the saved config. The Zod schema (Plan 01) ensures defaults for missing fields.
   - Also load processing config when constructing AppChannel objects from saved data.

10. **Modify `persistChannels`:**
    - Include processing config in the persisted channel data (agc, opus, rtpOutput, mode).

11. **Modify `stopChannel`:**
    - Clear any pending debounce timer for the channel: `this.clearDebouncedRestart(channelId)`.

12. **New private method: `clearDebouncedRestart(channelId: string): void`**
    - Clear and delete the timer from the debounce map if present.

13. **Cleanup in `removeChannel`:**
    - Call `clearDebouncedRestart(channelId)` before removal.

14. **Add to `ChannelUpdatableFields`:**
    - Do NOT add processing to ChannelUpdatableFields. Processing has its own dedicated method (`updateProcessingConfig`) with debounce logic. Keep it separate per SRP.

**ws/types.ts** -- Add processing WebSocket types:

1. Add to `ServerMessageType` union:
   - `"channel:processing:updated"` (broadcast when processing config changes)

2. New payload interfaces:
   ```typescript
   /** Request payload for channel:processing:update. */
   export interface ProcessingUpdatePayload {
     channelId: string;
     mode?: "speech" | "music";
     agc?: {
       enabled?: boolean;
       targetLufs?: number;
     };
     opus?: {
       enabled?: boolean;
       bitrateKbps?: number;
       frameSize?: "10" | "20" | "40";
       fec?: boolean;
       bitrateMode?: "vbr" | "cbr";
     };
   }

   /** Request payload for channel:processing:reset. */
   export interface ProcessingResetPayload {
     channelId: string;
   }

   /** Request payload for channel:processing:get. */
   export interface ProcessingGetPayload {
     channelId: string;
   }
   ```

   Note: rtpOutput is NOT exposed in the update payload -- port allocation is automatic (admin cannot manually set RTP ports). SSRC is auto-generated. This prevents admin from breaking port allocation.

**ws/handler.ts** -- Add processing message handlers:

1. Add `"processing:"` to `AUDIO_MESSAGE_PREFIXES` array so processing messages route to the audio handler.

2. Add cases in `handleAudioMessageAsync` switch:

   **`channel:processing:get`:**
   - Parse `ProcessingGetPayload`, validate channelId.
   - Call `audioSubsystem.getProcessingConfig(channelId)`.
   - Send response with processing config.

   **`channel:processing:update`:**
   - Parse `ProcessingUpdatePayload`, validate channelId.
   - Call `audioSubsystem.updateProcessingConfig(channelId, { mode, agc, opus })`.
   - Send response with updated channel.
   - The channel-updated event is already broadcast by the channel manager via existing wireAudioBroadcasts wiring (no new broadcast needed).

   **`channel:processing:reset`:**
   - Parse `ProcessingResetPayload`, validate channelId.
   - Call `audioSubsystem.resetProcessingDefaults(channelId)`.
   - Send response with updated channel.

3. Import new payload types from `./types`.

**Gain reduction in level broadcasts:**
No additional work needed here. The gainReductionDb field was added to NormalizedLevels in Plan 02. The existing level broadcast in wireAudioBroadcasts already serializes the full NormalizedLevels object, which now includes gainReductionDb. Admin clients will automatically receive it.
