**New internal functions (add to pipeline-builder.ts):**

1. `buildAgcChain(agc: AgcConfig): string`
   - If `agc.enabled === false`, return empty string (AGC bypassed)
   - Otherwise produce:
     ```
     audioconvert ! audioresample ! audio/x-raw,rate=192000 !
     audioloudnorm loudness-target={agc.targetLufs} max-true-peak={agc.maxTruePeakDbtp} !
     audioconvert ! audioresample ! audio/x-raw,rate=48000,channels=1 !
     ```
   - CRITICAL: audioresample wrappers around audioloudnorm are mandatory (192kHz internal requirement per research). Without them, audioloudnorm will crash or produce silence.
   - When AGC is bypassed, still force `audio/x-raw,rate=48000,channels=1` caps for downstream Opus compatibility.

2. `buildOpusRtpChain(opus: OpusEncodingConfig, rtp: RtpOutputConfig): string`
   - If `opus.enabled === false`, return empty string (encoding bypassed -- audio stays in metering-only mode)
   - Map bitrateMode: "vbr" -> `bitrate-type=constrained-vbr`, "cbr" -> `bitrate-type=cbr`
   - Map frameSize: string "20" -> number for `frame-size` property
   - Produce:
     ```
     opusenc bitrate={opus.bitrateKbps * 1000} frame-size={frameSize} audio-type={opus.audioType} bitrate-type={bitrateTypeGst} inband-fec={opus.fec} dtx=false !
     rtpopuspay pt=101 ssrc={rtp.ssrc} !
     rtpbin name=rtpbin_{rtp.ssrc} !
     udpsink host={rtp.host} port={rtp.rtpPort} sync=false
     rtpbin_{rtp.ssrc}.send_rtcp_src_0 ! udpsink host={rtp.host} port={rtp.rtcpPort} sync=false async=false
     ```
   - Use unique rtpbin name (include SSRC) to avoid GStreamer element name collisions if multiple pipelines somehow share a process space (defensive).
   - IMPORTANT opusenc audio-type naming: Speech mode -> `voice` (NOT "voip"), Music mode -> `generic` (NOT "audio"). Per research.

3. `buildProcessingAndOutputTail(processing: ProcessingConfig, levelIntervalNs: number): string`

   - Structure:

   ```

   ```
   [agcChain] level interval={levelIntervalNs} post-messages=true ! fakesink sync=false
   ```
   No tee needed -- metering-only after AGC.

   **Case C: AGC bypassed, Opus enabled (encoding without processing):**
   ```
   audioconvert ! audioresample ! audio/x-raw,rate=48000,channels=1 !

   ```
   Force mono 48kHz caps for Opus even without AGC.

   ```
   audioconvert ! audioresample ! level interval={levelIntervalNs} post-messages=true ! fakesink sync=false

   This is the original `buildMeteringTail` output.

**Gain reduction indicator** -- Per research, audioloudnorm does not expose current gain as a property. The approach: compute gain reduction by comparing consecutive level readings against a reference.

Since the `level` element in the Phase 3 pipeline sits AFTER the AGC stage (audioloudnorm -> tee -> level branch), the levels reflect post-AGC output. To compute gain reduction, we need a pre-AGC reference level. However, adding a second `level` element before audioloudnorm adds complexity and GStreamer overhead.

**Simplified approach (per research recommendation):** Approximate gain reduction using the difference between the current output RMS and a rolling "input estimate." Since we cannot directly measure pre-AGC level in a single `gst-launch-1.0` pipeline without adding a pre-AGC level element, use a simpler approach:

Actually, we CAN add a pre-AGC level element. The pipeline string builder can insert a `level` element with a different name before audioloudnorm. GStreamer emits level messages for ALL level elements in the pipeline, and we can distinguish them. But this is complex to parse (two level messages per frame).

**Best practical approach:** Add a pre-AGC `level` element named `prelevel` with a tag in the pipeline, and parse both level messages from stderr. The metering-parser already handles level messages -- we need to distinguish pre-AGC from post-AGC.

**Revised approach (simpler, avoids parser complexity):** The `level` element in GStreamer's `-m` mode does not include an element name in its output. So distinguishing two `level` elements by name is not straightforward from stderr. Instead:

**Final approach per research doc:** Compute `gainReductionDb` as an approximation: track the rolling max input level (from the first few seconds before AGC engages, or from a separate window). This is overly complex.

**Practical decision (Claude's discretion -- gain reduction indicator visual design):** Track gain reduction by storing a short-term "pre-normalization" reference level. Since we cannot easily get pre-AGC levels from a single gst-launch pipeline, compute an **estimated gain reduction** as: `targetLufs - currentRmsDb`. When AGC is working correctly, the output RMS should hover around the target LUFS. The gain reduction is the difference between what the AGC is "pushing toward" and the actual output. If output is at -16 LUFS (target), gain reduction is ~0. If the source was loud (-6 dB) and the output is -16, the implied compression is ~10 dB.

This is a display-only estimate suitable for an admin dashboard indicator. It does not need to be acoustically precise.

**level-monitor.ts changes:**

1. Add `gainReductionDb` field to `NormalizedLevels`:
   ```typescript
   readonly gainReductionDb: number;
   ```
   This represents estimated gain reduction in dB. Negative values mean compression is active (gain is being reduced). Zero means no compression. The admin dashboard will show this as a simple indicator.

2. Add a `processingTargets` map: `Map<string, { targetLufs: number }>` -- stores the per-pipeline AGC target LUFS. Set via a new method `setProcessingTarget(pipelineId: string, targetLufs: number)`.

3. In `handleLevels()`, compute `gainReductionDb`:
   - If no processing target set for this pipeline, `gainReductionDb = 0`
   - Otherwise: `gainReductionDb = avgRmsDb - targetLufs`
     - Where `avgRmsDb` is the average of rmsDb array values (across channels)
     - If avgRmsDb is -Infinity (silence), `gainReductionDb = 0`
   - This gives: positive = output louder than target (rare, transient), negative = output quieter than target (AGC settling), ~0 = AGC converged on target

4. Add `setProcessingTarget(pipelineId: string, targetLufs: number): void` method.
5. Add `clearProcessingTarget(pipelineId: string): void` -- called alongside `clearPipeline`.
6. Update `clearPipeline` to also call `clearProcessingTarget`.

**pipeline-types.ts** -- No additional changes needed beyond Plan 01 (processing field already added).
