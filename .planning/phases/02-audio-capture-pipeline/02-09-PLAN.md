    **Create audio-subsystem.ts:**

    The `AudioSubsystem` class is the top-level facade that wires all audio components together. External code (index.ts, ws handler) only interacts with this facade.

    **Imports:** All audio module classes (PipelineManager, DiscoveryManager, SourceRegistry, ChannelManager, LevelMonitor, ResourceMonitor, EventLogger)

    **Constructor:**
    ```typescript
    constructor(private configStore: ConfigStore, private basePath: string)
    ```
    - Create instances of all audio components:
      - `sourceRegistry = new SourceRegistry(basePath)`
      - `pipelineManager = new PipelineManager(configStore.get().audio.pipelineRecovery)`
      - `discoveryManager = new DiscoveryManager(sourceRegistry, { devicePollIntervalMs: configStore.get().audio.discoveryCache.devicePollIntervalMs })`
      - `levelMonitor = new LevelMonitor()`
      - `resourceMonitor = new ResourceMonitor()`
      - `eventLogger = new EventLogger(basePath)`
      - `channelManager = new ChannelManager(pipelineManager, sourceRegistry, levelMonitor, resourceMonitor, eventLogger, configStore)`

    **Method: `async start(): Promise<void>`**
    1. Start discovery (SAP listener, device enumeration polling)
    2. Start resource monitor
    3. Auto-start configured channels: `channelManager.startAll()`
    4. Log "Audio subsystem started"

    **Method: `async stop(): Promise<void>`**
    1. Stop all channels: `channelManager.stopAll()`
    2. Stop discovery: `discoveryManager.stop()`
    3. Stop resource monitor: `resourceMonitor.stop()`
    4. Flush event logger: `eventLogger.stop()`
    5. Log "Audio subsystem stopped"

    **Accessor methods (delegate to internal components):**
    - `getSources(): DiscoveredSource[]` -> sourceRegistry.getAll()
    - `getSourceById(id: string)` -> sourceRegistry.getById(id)
    - `getChannels(): AppChannel[]` -> channelManager.getAllChannels()
    - `getChannel(id: string)` -> channelManager.getChannel(id)
    - `createChannel(name, outputFormat?)` -> channelManager.createChannel(...)
    - `removeChannel(id)` -> channelManager.removeChannel(id)
    - `updateChannel(id, updates)` -> channelManager.updateChannel(...)
    - `addSource(channelId, assignment)` -> channelManager.addSource(...)
    - `removeSource(channelId, index)` -> channelManager.removeSource(...)
    - `updateSource(channelId, index, updates)` -> channelManager.updateSource(...)
    - `startChannel(id)` -> channelManager.startChannel(id)
    - `stopChannel(id)` -> channelManager.stopChannel(id)
    - `getLevels(pipelineId)` -> levelMonitor.getLevels(pipelineId)
    - `getAllLevels()` -> levelMonitor.getAllLevels()
    - `getStats(pipelineId)` -> resourceMonitor.getStats(pipelineId)
    - `getAllStats()` -> resourceMonitor.getAllStats()
    - `getChannelEvents(channelId, limit?)` -> eventLogger.getEvents(channelId, limit)
    - `getChannelPipelineIds(channelId)` -> channelManager.getChannelPipelineIds(channelId)

    **Event forwarding:**
    - Forward sourceRegistry "sources-changed" event
    - Forward channelManager "channel-created", "channel-updated", "channel-removed", "channel-state-changed" events
    - Forward levelMonitor "levels-updated" event
    - Forward resourceMonitor "stats-updated" event
    - All events are re-emitted on the AudioSubsystem instance so the WS handler only subscribes to one EventEmitter

    ```typescript
    export interface SourcesListPayload {
      sources: unknown[]; // DiscoveredSource[] -- use unknown to avoid circular dep, typed at runtime

    }

    }

    export interface ChannelUpdatePayload {
      channelId: string;
      name?: string;
      outputFormat?: "mono" | "stereo";
      autoStart?: boolean;
    }

    export interface ChannelSourceAddPayload {
      channelId: string;
      sourceId: string;
      selectedChannels: number[];
      gain?: number;
      muted?: boolean;
      delayMs?: number;
    }

    export interface ChannelSourceRemovePayload {
      channelId: string;
      sourceIndex: number;
    }

    export interface ChannelSourceUpdatePayload {
      channelId: string;
      sourceIndex: number;
      gain?: number;
      muted?: boolean;
      delayMs?: number;
      selectedChannels?: number[];
    }

    export interface ChannelActionPayload {
      channelId: string;
    }

    // Level data broadcast
    export interface LevelsPayload {
      levels: Record<string, unknown>; // pipelineId -> NormalizedLevels
    }

    // Resource stats
    export interface StatsPayload {
      stats: Record<string, unknown>; // pipelineId -> PipelineStats
    }

    // Channel events
    export interface ChannelEventsPayload {
      channelId: string;
      events: unknown[]; // ChannelEvent[]
    }
    ```

    **Extend ws/handler.ts:**

    The WebSocket handler needs to accept the AudioSubsystem as a dependency and handle new audio message types.

    1. **Update `setupWebSocket` signature** to accept an optional `audioSubsystem` parameter:
       ```typescript
       export function setupWebSocket(
         server: HttpServer | HttpsServer,
         configStore: ConfigStore,
         serverEvents: EventEmitter,
         audioSubsystem?: AudioSubsystem,
       ): WebSocketSetupResult
       ```
       If audioSubsystem is undefined (e.g., during early startup before audio is ready), audio message types return an error.

    2. **Add new cases to the message switch in `handleIncomingMessage`:**

       Admin-only messages (require role === "admin"):
       - `"sources:list"`: Call `audioSubsystem.getSources()`, respond with sources list
       - `"channels:list"`: Call `audioSubsystem.getChannels()`, respond with channels list
       - `"channel:create"`: Parse ChannelCreatePayload, call `audioSubsystem.createChannel(name, outputFormat)`, respond with created channel
       - `"channel:update"`: Parse ChannelUpdatePayload, call `audioSubsystem.updateChannel(...)`, respond with updated channel
       - `"channel:remove"`: Parse ChannelActionPayload, call `audioSubsystem.removeChannel(channelId)`, respond with ack
       - `"channel:source:add"`: Parse ChannelSourceAddPayload, call `audioSubsystem.addSource(...)`, respond with updated channel
       - `"channel:source:remove"`: Parse ChannelSourceRemovePayload, call `audioSubsystem.removeSource(...)`, respond with updated channel
       - `"channel:source:update"`: Parse ChannelSourceUpdatePayload, call `audioSubsystem.updateSource(...)`, respond with updated channel
       - `"channel:start"`: Parse ChannelActionPayload, call `audioSubsystem.startChannel(channelId)`, respond with ack
       - `"channel:stop"`: Parse ChannelActionPayload, call `audioSubsystem.stopChannel(channelId)`, respond with ack
       - `"channel:events"`: Parse ChannelActionPayload, call `audioSubsystem.getChannelEvents(channelId)`, respond with events
       - `"levels:subscribe"`: Flag this client for level updates (see broadcast below)
       - `"stats:get"`: Respond with current stats for all pipelines

       Each handler should:
       - Check admin role (return unauthorized for non-admin)
       - Wrap in try/catch, return error payload on failure
       - Use the existing `sendMessage` helper for responses

    3. **Level data broadcasting:**
       Subscribe to audioSubsystem "levels-updated" event. On each update, broadcast to all admin clients that have the WebSocket open. Throttle broadcasts to avoid flooding:
       - Collect level updates in a buffer
       - Flush buffer every 100ms (or 33ms if configured) to all admin clients as a batch
       - Use `setInterval` for the flush timer
       - Only send if buffer has data (don't send empty updates)
       - The batch payload is `{ type: "levels:update", payload: { levels: { [pipelineId]: normalizedLevels } } }`

    4. **Source change broadcasting:**
       Subscribe to audioSubsystem "sources-changed" event. Broadcast `{ type: "sources:changed" }` to all admin clients (they can then request the full list via "sources:list").

    5. **Channel change broadcasting:**
       Subscribe to audioSubsystem channel events. Broadcast to all admin clients:
       - "channel:created" -> on channel-created
       - "channel:updated" -> on channel-updated
       - "channel:removed" -> on channel-removed
       - "channel:state" -> on channel-state-changed

    **Important:** The handler.ts is already large. Keep the new audio handlers as a separate function `handleAudioMessage(socket, message, audioSubsystem)` called from the main switch when message.type starts with "sources:", "channel:", "channels:", "levels:", or "stats:". This follows SRP -- the main handler routes, the audio handler processes.
