    **Create level-monitor.ts:**

    `LevelMonitor` class that aggregates audio level data from all pipelines and provides it for UI consumption.

    **Imports:**
    - `EventEmitter` from `node:events`
    - `AudioLevels` from `../pipeline/pipeline-types`
    - `dbToNormalized` from `../pipeline/metering-parser`

    **Internal state:**
    - `latestLevels: Map<string, NormalizedLevels>` -- keyed by pipeline ID, stores latest normalized level data
    - `clippingState: Map<string, { clipping: boolean, clearedAt: number }>` -- tracks clip indicator per pipeline

    **Type: `NormalizedLevels` (exported):**
    - `pipelineId: string`
    - `peak: number[]` -- 0.0-1.0 range per channel
    - `rms: number[]` -- 0.0-1.0 range per channel
    - `peakDb: number[]` -- raw dB values per channel
    - `rmsDb: number[]` -- raw dB values per channel
    - `clipping: boolean` -- momentary clip indicator
    - `timestamp: number`

    **LevelMonitor extends EventEmitter with events:**
    - `"levels-updated"` -> (levels: NormalizedLevels)

    **Method: `handleLevels(pipelineId: string, rawLevels: AudioLevels): void`**
    - Convert raw dB values to 0.0-1.0 via dbToNormalized for each channel
    - Create NormalizedLevels object
    - If rawLevels.clipping is true, set clipping state for this pipeline (momentary -- CONTEXT.md says "momentary red flash")
    - Store in latestLevels map
    - Emit "levels-updated"

    **Method: `getLevels(pipelineId: string): NormalizedLevels | undefined`**
    - Return latest levels for a pipeline

    **Method: `getAllLevels(): Map<string, NormalizedLevels>`**
    - Return copy of all latest levels

    **Method: `clearPipeline(pipelineId: string): void`**
    - Remove from latestLevels and clippingState maps

    ---

    **Create resource-monitor.ts:**

    `ResourceMonitor` class that polls pidusage for CPU% and memory per GStreamer pipeline process.

    **Imports:**
    - `EventEmitter` from `node:events`
    - pidusage (default import -- `import pidusage from "pidusage"`)
    - `PipelineStats` from `../pipeline/pipeline-types`
    - `logger` from `../../utils/logger`

    **Internal state:**
    - `trackedPids: Map<string, { pid: number, startedAt: number }>` -- pipelineId -> PID + start time
    - `latestStats: Map<string, PipelineStats>` -- pipelineId -> latest resource stats
    - `pollInterval: NodeJS.Timeout | null`
    - `pollIntervalMs: number` (default 5000 -- 5 seconds between resource checks)

    **ResourceMonitor extends EventEmitter with events:**
    - `"stats-updated"` -> (pipelineId: string, stats: PipelineStats)

    **Constructor:** `constructor(pollIntervalMs: number = 5000)`

    **Method: `trackPipeline(pipelineId: string, pid: number): void`**
    - Add to trackedPids map with current timestamp

    **Method: `untrackPipeline(pipelineId: string): void`**
    - Remove from trackedPids and latestStats

    **Method: `start(): void`**
    - Start polling interval
    - On each tick: for each tracked PID, call `pidusage(pid)`:
      - pidusage returns `{ cpu: number, memory: number, elapsed: number }`
      - Convert to PipelineStats: `cpuPercent = result.cpu`, `memoryMb = result.memory / (1024 * 1024)`, `uptimeMs = Date.now() - startedAt`, `pid`
      - Store in latestStats, emit "stats-updated"
    - Handle errors gracefully: if PID no longer exists, untrack it (process may have exited)

    **Method: `stop(): void`**
    - Clear interval, set pollInterval to null

    **Method: `getStats(pipelineId: string): PipelineStats | undefined`**
    - Return latestStats.get(pipelineId)

    **Method: `getAllStats(): Map<string, PipelineStats>`**
    - Return copy of all latest stats

    **Important:**
    - pidusage on Windows uses wmic under the hood, which can be slow. 5-second polling avoids overloading.
    - If the tracked PID is no longer valid (process exited), pidusage throws. Catch this and untrack.
    - CPU% from pidusage is percentage of one core (0-100+ on multi-core). Display as-is.
  </action>
  <verify>
    `npx tsc --noEmit` passes. LevelMonitor exports with handleLevels(), getLevels(), getAllLevels(). ResourceMonitor exports with trackPipeline(), start(), stop(), getStats(). Verify pidusage import compiles.
  </verify>
  <done>
    LevelMonitor converts raw GStreamer dB levels to normalized 0-1 range with clipping detection. ResourceMonitor polls pidusage for per-pipeline CPU% and memory. Both provide getter methods and emit events for downstream WebSocket broadcast.

  <name>Task 2: Create EventLogger with disk persistence and retention</name>
  <files>sidecar/src/audio/monitor/event-logger.ts</files>
  <action>
    Create `event-logger.ts` that exports an `EventLogger` class for per-channel event logging with disk persistence.

    **Type: `ChannelEvent` (exported):**
    - `channelId: string`
    - `timestamp: number` (Unix timestamp ms)
    - `type: "start" | "stop" | "crash" | "restart" | "source-change" | "error" | "warning" | "info"`
    - `message: string` (human-readable)
    - `details?: Record<string, unknown>` (optional structured data for expandable technical details)

    **Imports:**
    - `fs` from `node:fs`
    - `path` from `node:path`
    - `logger` from `../../utils/logger`

    **Internal state:**
    - `logDirectory: string` -- base directory for channel event logs
    - `events: Map<string, ChannelEvent[]>` -- in-memory cache per channel, keyed by channelId
    - `maxEventsInMemory: number` (default 1000 per channel -- keep recent events in RAM for quick access)
    - `retentionDays: number` (default 30)
    - `flushTimer: NodeJS.Timeout | null` -- debounced flush to disk

    **Constructor:** `constructor(basePath: string, retentionDays: number = 30)`
    - Set logDirectory to `path.join(basePath, "logs", "channels")`
    - Create directory if it doesn't exist (recursive)
    - Load existing event logs from disk for all channels
    - Run retention cleanup (delete events older than retentionDays)

    **Method: `log(event: ChannelEvent): void`**
    - Add to in-memory events map for the channel
    - Trim in-memory array if > maxEventsInMemory (oldest first)
    - Schedule debounced flush to disk (500ms debounce)
    - Also log via logger.info/warn/error based on event type

    **Method: `getEvents(channelId: string, limit?: number): ChannelEvent[]`**
    - Return events for a channel, most recent first
    - If limit specified, return only that many

    **Method: `getRecentEvents(channelId: string, sinceMs: number): ChannelEvent[]`**
    - Return events newer than sinceMs timestamp

    **Method: `clearChannel(channelId: string): void`**
    - Remove all events for a channel from memory and disk

    **Disk persistence (private):**
    - Each channel gets its own file: `{logDirectory}/{channelId}.jsonl` (JSON Lines format -- one JSON object per line)
    - `flushToDisk(channelId: string)`: Append new events to the channel's JSONL file (not rewrite the whole file)
    - `loadFromDisk(channelId: string)`: Read the JSONL file, parse each line, populate in-memory cache
    - `loadAllChannels()`: List files in logDirectory, load each
    - Use JSONL (newline-delimited JSON) because: append-only writes are cheap, easy to parse line by line, tolerant of partial writes

    **Retention cleanup (private):**
    - `runRetentionCleanup()`: For each channel log file:
      - Read all lines
      - Filter out events older than retentionDays
      - Rewrite the file with remaining events
      - Update in-memory cache
    - Run once on startup
    - Schedule periodic cleanup (every 24 hours via setInterval)

    **Method: `stop(): void`**
    - Flush all pending events to disk immediately
    - Clear retention cleanup timer

    **Important:**
    - JSONL format is append-only friendly and crash-safe (partial last line can be discarded on load)
    - CONTEXT.md specifies 30-day retention and events persisted to disk
    - Events include: start, stop, crash, restart, source-change, error, warning, info
    - On load, skip malformed JSON lines (log warning, continue)
    - Channel ID in filename must be filesystem-safe (UUID is safe)
  </action>

  <done>
    EventLogger provides per-channel event logging with JSONL disk persistence. Events survive restarts. 30-day retention with automatic cleanup. Debounced writes prevent excessive disk I/O. In-memory cache provides fast access for recent events.
