<objective>
Implement local audio device enumeration using GStreamer's device monitor tool, with polling-based hot-plug detection.

Purpose: Admins need to see available audio input devices (USB interfaces, microphones, line-in, virtual devices) categorized by Windows audio API (WASAPI, ASIO, DirectSound). The device enumerator runs gst-device-monitor-1.0 and parses the JSON output, filtering out Bluetooth devices and flagging WASAPI loopback capabilities.

Output: `device-enumerator.ts` provides a `DeviceEnumerator` class that discovers local audio devices, detects hot-plug changes via polling, and emits events for device-added/device-removed.

  <action>
    Create `device-enumerator.ts` that exports a `DeviceEnumerator` class for discovering local audio input devices.

    **Types (exported):**

    `EnumeratedDevice`:
    - `id: string` -- unique ID composed of `${api}:${deviceId}` (same device appears separately per API)
    - `name: string` -- display name (e.g., "Focusrite USB Audio")
    - `api: "wasapi2" | "asio" | "directsound"` -- which GStreamer plugin/API
    - `deviceId: string` -- the device identifier used in GStreamer pipeline (wasapi2 device path, ASIO CLSID, DirectSound device name)
    - `sampleRate: number` -- default/native sample rate (0 if unknown)
    - `bitDepth: number` -- bit depth (0 if unknown)
    - `channelCount: number` -- number of input channels (0 if unknown)
    - `isLoopback: boolean` -- true for WASAPI loopback devices
    - `isBluetooth: boolean` -- true for Bluetooth devices (for filtering)

    **DeviceEnumerator extends EventEmitter with events:**
    - `"device-added"` -> (device: EnumeratedDevice)
    - `"device-removed"` -> (deviceId: string)
    - `"enumeration-complete"` -> (devices: EnumeratedDevice[])
    - `"error"` -> (error: Error)

    **Internal state:**
    - `currentDevices: Map<string, EnumeratedDevice>` -- keyed by device.id
    - `pollInterval: NodeJS.Timeout | null`
    - `pollIntervalMs: number`
    - `running: boolean`

    **Constructor:** `constructor(pollIntervalMs: number = 5000)` -- default 5s polling per RESEARCH recommendation

    **Method: `enumerate(): Promise<EnumeratedDevice[]>`**
    One-shot enumeration. Runs gst-device-monitor-1.0 and returns parsed devices.

    Implementation:
    1. Run `execFile('gst-device-monitor-1.0', ['Audio/Source', '-f', 'json'])` with a 10-second timeout
    2. Parse the JSON stdout output
    3. Map each device entry to an EnumeratedDevice:
       - Determine API from the device's plugin/factory name:
         - Factory contains "wasapi2" -> api = "wasapi2"
         - Factory contains "asio" -> api = "asio"
         - Factory contains "directsound" -> api = "directsound"
         - Other -> skip (unknown plugin)
       - Extract device ID from properties:
         - wasapi2: `device.id` or `device.path` property (the Windows device endpoint path)
         - ASIO: `device.device-clsid` property (GUID string)
         - DirectSound: `device.device-name` property
       - Extract audio capabilities from device caps string:
         - Parse `audio/x-raw` caps for rate, format (S16LE, S24LE, S32LE, F32LE -> bit depth), channels
         - Take the first (preferred) caps entry
       - Detect loopback: wasapi2 devices with `device.loopback` property = true, or device name containing "loopback" or "monitor" (case-insensitive)
       - Detect Bluetooth: check device name for "bluetooth", "bt", or device path containing "BTH" or "Bluetooth"
    4. Filter: EXCLUDE devices where `isBluetooth === true` (CONTEXT.md: Bluetooth excluded)
    5. Also run `gst-device-monitor-1.0 Audio/Sink -f json` to enumerate output devices (for the audio monitor feature: admin can route sources to output devices). Parse these similarly but mark them as sinks. Store sinks separately if needed, OR return them in a separate method `enumerateOutputDevices()`.
    6. Return the filtered array, emit "enumeration-complete"

    **NOTE on gst-device-monitor JSON format:** The exact JSON structure from gst-device-monitor-1.0 may vary. The implementation should be defensive:
    - If the output is not valid JSON, try to detect if it's the human-readable format instead and log a warning
    - If `-f json` flag is not supported (older GStreamer), fall back to text parsing with a clear log message
    - Structure typically: array of device objects with `name`, `device-class`, `properties` (key-value), `caps` (GstCaps string)

    **Method: `startPolling(): void`**
    Start periodic polling for hot-plug detection.
    - Run `enumerate()` on interval
    - Compare new device list to `currentDevices`:
      - Devices in new list but not in current: emit "device-added" for each
      - Devices in current but not in new list: emit "device-removed" for each
    - Update `currentDevices`
    - Per RESEARCH: polling is more reliable than GStreamer DeviceMonitor events on Windows

    **Method: `stopPolling(): void`**
    - Clear the interval timer
    - Set running = false

    **Method: `getCurrentDevices(): EnumeratedDevice[]`**
    - Return Array.from(currentDevices.values())

    **Method: `getDeviceById(id: string): EnumeratedDevice | undefined`**
    - Lookup from currentDevices map

    **Error handling:**
    - If gst-device-monitor-1.0 is not found (ENOENT), emit "error" with message "GStreamer not found. Install GStreamer 1.26 and add to PATH."
    - If JSON parse fails, emit "error" and return empty array (don't crash)
    - Polling errors should log and continue (don't stop polling on transient errors)

    **Important per CONTEXT.md:**
    - Show ALL devices per API: same physical device appears as both "Focusrite (WASAPI)" and "Focusrite (ASIO)"
    - Show virtual devices (VB-Cable, VoiceMeeter) -- don't filter them
    - WASAPI loopback capture is supported (system audio output)
    - Bluetooth devices are EXCLUDED
