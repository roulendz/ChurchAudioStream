Fix the blocker preventing Tauri admin UI from connecting to the sidecar. The root cause is that the server is HTTPS-only, and Tauri WebView2 rejects the self-signed TLS certificate during the wss:// handshake with no way to accept it.

Purpose: Without this fix, the admin UI is completely non-functional -- stuck on "Connecting..." forever.

Output: A dual-listener server where Tauri connects via plain ws://localhost:PORT (safe on loopback, no TLS cert issues) while phone browsers connect via wss://IP:PORT (encrypted for WiFi network).
</objective>

2. Update the `ServerComponents` interface:
   - Rename `server` to `httpsServer` (type: `https.Server`)
   - Add `httpServer` field (type: `http.Server`)
   - Keep all other fields (`app`, `wss`, `getClients`, `key`, `cert`)
   - Add a new `httpWss` field for the loopback WebSocketServer setup result

3. In `createServer()`:
   - After creating `httpsServer` and calling `setupWebSocket(httpsServer, ...)`, also:
   - Create a plain HTTP server: `const httpServer = http.createServer(app);`
   - Call `setupWebSocket(httpServer, configStore, serverEvents)` to get a second `{ wss, getClients }` for the HTTP server.
   - Store the HTTP WSS result separately. Name it `httpWsSetup` vs `httpsWsSetup`.
   - Return both servers and both WSS setups in `ServerComponents`.

4. In `startServer()`:
   - Listen the HTTPS server on `config.server.listenHost` (0.0.0.0) + `config.server.port` as before.
   - Listen the HTTP server on `127.0.0.1` + SAME port. This is safe because HTTPS binds to 0.0.0.0 (all interfaces) which includes 127.0.0.1, BUT Node.js allows a second server on the same port with a specific IP bind as long as one is specific and the other is 0.0.0.0.
   - IMPORTANT: Actually, binding two servers to the same port (even with different IPs) may fail on some OSes. Instead, bind the HTTP server to port `config.server.port + 1` as the loopback port. Update log messages to show both ports.
   - Wait: Re-evaluate. On Windows, binding 0.0.0.0:7777 and 127.0.0.1:7777 simultaneously WILL conflict because 0.0.0.0 includes 127.0.0.1. Use a fixed offset: HTTP loopback port = `config.server.port + 1` (e.g., 7778).
   - Add the HTTP server listen call: `httpServer.listen(config.server.port + 1, "127.0.0.1", callback)`.
   - Update logging to show both listeners:
     ```
     logger.info("HTTPS server listening (external)", { address: `0.0.0.0:${port}` });
     logger.info("HTTP server listening (loopback)", { address: `127.0.0.1:${port + 1}` });
     ```

5. In `stopServer()` (the returned closure):
   - Broadcast restart message on BOTH WebSocketServers before closing.
   - Close BOTH servers: `httpsServer.close()` and `httpServer.close()`.
   - Include the loopback port in the restart payload so the frontend knows the new loopback port too.

6. Update `broadcastAndClose` to iterate over both WSS instances' clients.
