    **Server restart logic (`sidecar/src/server.ts`):**
    - Add an exported `restartServer` function that:
      1. Sends "server:restarting" message to all connected WS clients with the new host/port info
      2. Closes all WebSocket connections with close code 1012 ("Service Restart")
      3. Calls the stopServer function (closes HTTPS server, unpublishes mDNS, waits for 'close' event)
      4. Waits 500ms for port release (Pitfall 6: TCP TIME_WAIT)
      5. Creates a NEW HTTPS server with the updated config (re-reads cert, creates new Express app)
      6. Creates a NEW WebSocketServer attached to the new HTTPS server
      7. Calls startServer with new config
      8. Returns the new server/wss instances
    - This function needs access to the ConfigStore to re-read current config after update
    - Handle EADDRINUSE: if the new port is occupied, log error, attempt to restart on the OLD port as fallback, notify clients of failure

    **Concrete EventEmitter wiring:**

    The serverEvents EventEmitter was created in index.ts and passed to setupWebSocket in Plan 02. Now we wire the two ends:

    1. In `sidecar/src/ws/handler.ts` (the EMITTER side):
       - The setupWebSocket function already receives `serverEvents: EventEmitter` as a parameter
       - Inside the "config:update" message handler, after calling configStore.update() and detecting that server.port, server.host, or server.interface changed:
         ```typescript
         // Send response to client first
         sendResponse(ws, { type: 'config:updated', payload: { config: newConfig, requiresRestart: true }, requestId });
         // Broadcast to all admin clients
         broadcastToAdmins({ type: 'config:updated', payload: { config: newConfig, requiresRestart: true } });
         // Emit restart event after 1-second delay so response reaches clients
         setTimeout(() => {
           serverEvents.emit('restart-needed');
         }, 1000);
         ```
       - The handler does NOT import server.ts or call restartServer directly (SRP)

    2. In `sidecar/src/index.ts` (the LISTENER side):
       - After creating the server and calling startServer, register the restart listener:
         ```typescript
         let currentStopServer: (() => Promise<void>) | null = stopFn;

         serverEvents.on('restart-needed', async () => {
           logger.info('Server restart requested due to config change');
           try {
             if (currentStopServer) await currentStopServer();
             await new Promise(resolve => setTimeout(resolve, 500)); // TCP TIME_WAIT
             const config = configStore.get();
             const { server: newServer, wss } = await createServer(config, basePath, configStore, serverEvents);
             const newStopFn = await startServer(newServer, config);
             currentStopServer = newStopFn;
             logger.info(`Server restarted on https://${config.server.host}:${config.server.port}`);
           } catch (err) {
             logger.error('Failed to restart server', { error: String(err) });
             // Attempt to restart on old config as fallback
           }
         });
         ```
       - Note: Because createServer passes serverEvents through to setupWebSocket, the new WebSocket handler automatically gets the same EventEmitter. No need to re-register the listener -- it persists on the EventEmitter instance.

    **Add "interfaces:list" WebSocket message type:**
    - In handler.ts, add handler for "interfaces:list" message:
      - Calls listNetworkInterfaces()
      - Returns array of available interfaces with name and address
      - Available to all roles (admin and listener may both need this for diagnostics)

    IMPORTANT: Server restart must be sequential (close old, wait, start new). Never have two servers trying to bind simultaneously.
    IMPORTANT: After restart, all clients will need to reconnect. The admin UI handles this via auto-reconnect (Task 2).
    IMPORTANT: The WebSocket handler should NOT directly import or call server functions. Use the serverEvents EventEmitter pattern described above to keep separation of concerns (SRP).

    **useWebSocket hook (`src/hooks/useWebSocket.ts`):**
    - Manages a single WebSocket connection to the sidecar
    - Constructor: takes URL (wss://localhost:{port})
    - Auto-reconnect with exponential backoff: 1s, 2s, 4s, 8s, max 30s
    - On connect: send "identify" message with role "admin"
    - Expose:
      - `status`: 'connecting' | 'connected' | 'disconnected' | 'reconnecting'
      - `sendMessage(type, payload?, requestId?)`: send a typed WsMessage
      - `subscribe(type, callback)`: register handler for specific message types
      - `lastMessage`: most recent message received (for debugging)
    - On "server:restarting" message: note the new port from payload, update reconnect URL
    - On close with code 1012: immediately start reconnect (server restart, not error)
    - On unexpected close: start exponential backoff reconnect
    - Cleanup on unmount: close WebSocket, cancel reconnect timers

    **useServerStatus hook (`src/hooks/useServerStatus.ts`):**
    - Uses useWebSocket internally
    - On connect, sends "config:get" to load current config
    - Subscribes to "config:updated" for live updates
    - Subscribes to "server:status" for status info
    - Exposes:
      - `config`: current AppConfig (or null if not loaded)
      - `connectionStatus`: from useWebSocket
      - `updateConfig(partial)`: sends config:update message, returns promise that resolves with result
      - `interfaces`: list of available network interfaces (fetched on mount via "interfaces:list")

    **ConnectionStatus component (`src/components/ConnectionStatus.tsx`):**
    - Always visible at the top of the admin UI (sticky/fixed position)
    - Shows colored indicator dot + text:
      - Green dot + "Connected" when connected
      - Yellow dot + "Reconnecting..." when reconnecting (with attempt count)
      - Red dot + "Disconnected" when disconnected
    - Compact design -- should not take much vertical space
    - This fulfills the user's requirement: "Always-visible connection status indicator"

    **SettingsPanel component (`src/components/SettingsPanel.tsx`):**
    - Shows current server settings in a form:
      - Port: number input (1024-65535)
      - Network Interface: dropdown populated from interfaces list (show "name (address)" format)
      - mDNS Enabled: toggle/checkbox
      - mDNS Domain: text input (e.g., "churchaudio.local")
    - Save/Apply button workflow:
      - Form starts in "clean" state matching server config
      - Editing any field marks the form as "dirty" (Save button becomes active)
      - Clicking Save sends config:update via WebSocket
      - If response has requiresRestart, show a brief "Server restarting..." message
      - On success: show "Settings saved" confirmation
      - On validation error: show error messages next to relevant fields
    - DO NOT make this a full settings page -- just the Phase 1 config fields. Phase 6 adds the full dashboard.
    - Use controlled inputs with local state, only send to server on Save

    **LogViewer component (`src/components/LogViewer.tsx`):**
    - Listens to Tauri "sidecar-log" and "sidecar-error" events (from the Tauri event system)
    - Import `listen` from `@tauri-apps/api/event`
    - Displays log lines in a scrollable container (max height, auto-scroll to bottom)
    - Color-code by level: info=white, warn=yellow, error=red
    - Parse the JSON log format from the sidecar: extract level, timestamp, message
    - Keep a rolling buffer of last 200 log lines (prevent memory growth during long sessions)
    - Show a "Clear" button to empty the log viewer
    - This fulfills the user's requirement: "Admin can diagnose sidecar issues by viewing real-time logs"
    - IMPORTANT: Also listen for WebSocket messages for when running in browser mode (not Tauri). Check if Tauri API is available; if not, fall back to subscribing to a "logs:subscribe" WS message type.

    **App.tsx and App.css:**
    - Simple layout with three sections stacked vertically:
      1. ConnectionStatus (sticky top bar)
      2. SettingsPanel (main content area)
      3. LogViewer (bottom panel, collapsible)
