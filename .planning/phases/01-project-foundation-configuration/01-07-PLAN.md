Rewrite `ensureFirewallRule` in `sidecar/src/network/firewall.ts` to be more robust:

1. **Add elevation detection function** `isRunningElevated()`:
   - Run `net session` via `execSync` in a try/catch. If it succeeds (exit code 0), the process has admin privileges. If it throws, it does not.
   - This is the standard Windows technique for checking elevation.

2. **Improve the warning message when rule creation fails:**
   - If NOT elevated, log a warning with the EXACT netsh command the user can copy-paste into an elevated terminal:
     ```
     logger.warn("Firewall rule requires administrator privileges. Phone connections may be blocked.", {
       port,
       manualCommand: `netsh advfirewall firewall add rule name="${FIREWALL_RULE_NAME}" dir=in action=allow protocol=TCP localport=${port} profile=private,domain`,
       instructions: "Open PowerShell as Administrator and run the command above, OR right-click the app and choose 'Run as administrator' for one-time setup.",
     });
     ```
   - If elevated but rule creation still fails (rare -- maybe netsh is broken), log a different message with the specific error.

3. **Add a verification step after rule creation:**
   - After `createFirewallRule(port)` succeeds, call `firewallRuleMatchesPort(port)` to confirm the rule was actually created.
   - If verification fails, warn that the rule may not have been applied correctly.

4. **Keep the existing best-effort pattern** -- never throw, never crash. The function must always resolve, even on failure.

5. **Keep the existing functions** (`firewallRuleMatchesPort`, `deleteExistingRule`, `createFirewallRule`) -- they work correctly. Only enhance the orchestration in `ensureFirewallRule` and add `isRunningElevated`.
