<task type="auto">
  <name>Task 2: Add best-effort Windows Firewall rule on startup</name>
  <files>
    sidecar/src/network/firewall.ts
    sidecar/src/index.ts
  </files>
  <action>
    **Problem:** Even with the server bound to 0.0.0.0, Windows Firewall silently drops inbound TCP connections on non-standard ports. Phone browsers get ERR_EMPTY_RESPONSE because the packet never reaches Node.js.

    **Create `sidecar/src/network/firewall.ts`:**
    - Export an async function `ensureFirewallRule(port: number): Promise<void>`
    - Only runs on Windows (`process.platform === "win32"`). On other platforms, return immediately (no-op).
    - Use `child_process.execSync` to run:
      ```
      netsh advfirewall firewall show rule name="ChurchAudioStream"
      ```
      If the rule already exists AND the port matches, skip creation (idempotent).
    - If rule does not exist or port differs, run:
      ```
      netsh advfirewall firewall delete rule name="ChurchAudioStream"
      netsh advfirewall firewall add rule name="ChurchAudioStream" dir=in action=allow protocol=TCP localport={port} profile=private,domain
      ```
    - Wrap everything in try/catch. If the command fails (e.g., not running as administrator), log a warning:
      ```typescript
      logger.warn("Could not create Windows Firewall rule. Phone connections may be blocked.", {
        port,
        hint: "Run the application as Administrator, or manually add a firewall rule for TCP port " + port,
        error: errorMessage,
      });
      ```
    - NEVER crash or throw on failure. This is strictly best-effort.
    - Use the `logger` import from `../utils/logger`.
    - Note: Use `profile=private,domain` (not `public`) because church WiFi networks are typically private/domain networks. If the user's network is classified as "public" in Windows, they'll need to add the rule manually or change their network profile. The warning message covers this.

    **Update `sidecar/src/index.ts`:**
    - Import `ensureFirewallRule` from `./network/firewall`.
    - In the `main` function, call `await ensureFirewallRule(config.server.port)` AFTER config is loaded but BEFORE `startServer` is called (around line 160, after the config log and before `createServer`).
    - Also call `ensureFirewallRule` in the restart listener (`setupRestartListener`) after the new config is loaded and before the server restarts, in case the port changed. Add it inside the try block, before `createServer`.

    **IMPORTANT:** Do not import `child_process` at the top level in a way that would break non-Windows builds. The function body checks `process.platform` first, so the `execSync` import is fine at the module level since Node.js has `child_process` on all platforms.
  </action>
  <verify>
    - On Windows, start sidecar with admin privileges: right-click terminal -> Run as Administrator, then `cd sidecar && npx tsx src/index.ts -- --config-path .`
    - Check logs for "Firewall rule created" or similar success message
    - Verify rule exists: run `netsh advfirewall firewall show rule name="ChurchAudioStream"` in another terminal
    - Start sidecar WITHOUT admin privileges -- verify it logs a warning but does NOT crash
    - From a phone on the same WiFi, `https://{computer-IP}:7777` should now be reachable (after accepting self-signed cert)
  </verify>
  <done>Windows Firewall rule for the configured port is created on startup when running with admin privileges. When running without admin, a clear warning is logged with instructions. The rule is updated on server restart if the port changes. Non-Windows platforms skip this entirely.</done>
