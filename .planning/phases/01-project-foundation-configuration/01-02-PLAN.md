      - `server`: port (int, 1024-65535, default 7777), host (string, default from first non-loopback IPv4 interface), interface (string, optional -- name of network interface to bind)
      - `network.mdns`: enabled (bool, default true), domain (string, default "churchaudio.local")

    - Export `ConfigSchema` (the Zod object) and `AppConfig` type (z.infer)

    **Network Interfaces (`sidecar/src/network/interfaces.ts`):**
    - Export `listNetworkInterfaces()`: Returns array of {name, address, family, mac, internal}
      - Filter to IPv4, non-internal only
      - Use `os.networkInterfaces()` from Node.js
    - Export `getDefaultInterface()`: Returns first non-loopback IPv4 interface or undefined
    - Follow research code example exactly
    - NOTE: This file is created here (not Task 2) because defaults.ts needs getDefaultInterface

    **Config Defaults (`sidecar/src/config/defaults.ts`):**
    - Import ConfigSchema
    - Export `defaultConfig` as `ConfigSchema.parse({})` -- lets Zod apply all defaults
    - Export a `getDefaultHost` function that returns the first non-loopback IPv4 address (import from interfaces.ts)

    **ConfigStore (`sidecar/src/config/store.ts`):**
    - Class with constructor(basePath: string) that resolves config.json path
    - The basePath parameter is the directory where config.json lives. In pkg compiled binary mode, this MUST be path.dirname(process.execPath). In dev mode, this comes from --config-path argument. The ConfigStore itself does NOT decide where config lives -- the caller passes basePath.
    - `load()`: Read file, JSON.parse, safeParse with Zod. On invalid: log warning, reset to defaults, save defaults. On missing/corrupt JSON: same behavior. Service NEVER fails to start.
    - `get()`: Return deep clone (structuredClone) of current config
    - `update(partial)`: Deep merge partial into current config, safeParse result, save if valid, return {success, config, errors?}
    - `save(config)`: Write JSON with 2-space indent to disk
    - `getPath()`: Return the config file path (for logging)
    - Follow research Pattern 5 closely but use deep merge (not shallow spread) for nested config sections
    - Use synchronous fs operations for simplicity (config is read once at startup, written rarely)

    IMPORTANT: Do NOT bind to 0.0.0.0 by default. The user explicitly decided against this. Default host should be the first non-loopback IPv4 address from getDefaultInterface().
    IMPORTANT: Config path is resolved relative to basePath (which comes from process.execPath directory in production), NOT __dirname or import.meta.dirname. This ensures the config file lives next to the executable, not inside the pkg snapshot filesystem (Pitfall 4).
  </action>

    **Certificate (`sidecar/src/network/certificate.ts`):**
    - Export `loadOrGenerateCert(basePath: string, config: AppConfig)`:
      - Check if cert.pem and key.pem exist at basePath
      - If exist: read and return {key, cert}
      - If not: generate using `selfsigned` (async!) with:
        - commonName: config.network.mdns.domain
        - SANs: DNS for domain + "localhost", IP for all non-loopback IPv4 addresses + 127.0.0.1
        - keySize: 2048, days: 3650 (10 years), algorithm: sha256
      - Save generated cert.pem and key.pem to basePath
      - Return {key, cert}
    - Follow research code example for generateCertificate closely

    **mDNS (`sidecar/src/network/mdns.ts`):**
    - Export `publishService(port, domain)`: Create Bonjour instance, publish _http._tcp service
    - Export `unpublishService()`: Clean teardown of Bonjour instance
    - Include txt record with path "/" and protocol "https"
    - Follow research code example exactly

    **WebSocket Types (`sidecar/src/ws/types.ts`):**
    - Define the message format (research Pattern 4):
      ```typescript
      interface WsMessage {
        type: string;           // e.g., "identify", "config:get", "config:update"
        payload?: unknown;
        requestId?: string;     // For request/response correlation
      }

      interface IdentifyPayload {
        role: 'admin' | 'listener';
        clientId?: string;
      }
      ```
    - Define server-to-client message types:
      - "welcome": sent on connection with server version and assigned clientId
      - "error": sent when a message cannot be processed
      - "pong": response to "ping" (application-level, in addition to WS protocol ping)
    - Export all types

    **WebSocket Handler (`sidecar/src/ws/handler.ts`):**
    - Export `setupWebSocket(server: https.Server, configStore: ConfigStore, serverEvents: EventEmitter)` function:
      1. Create WebSocketServer attached to the HTTPS server (not a separate port)
      2. Set up heartbeat (research Pattern: WebSocket Heartbeat):
         - Ping every 30 seconds
         - If no pong within 10 seconds, terminate connection
         - Mark connections as alive on pong
      3. On 'connection':
         - Generate a unique clientId (crypto.randomUUID())
         - Send "welcome" message with { version, clientId }
         - Set a 10-second timeout for "identify" message -- if no identify received, close with code 4001
         - Store client metadata: { role, clientId, connectedAt, isAlive }
      4. On 'message':
         - Parse JSON (catch parse errors, send "error" response)
         - Route by message type:
           - "identify": Validate payload as IdentifyPayload, store role, acknowledge
           - "ping": Respond with "pong" (application-level keepalive)
           - "config:get": Return current config (admin only -- check role)
           - "config:update": Validate and apply config update (admin only), broadcast "config:updated" to all admin clients. Compare old vs new config -- if server.port, server.host, or server.interface changed, set `requiresRestart: true` in the response and emit `serverEvents.emit('restart-needed')` after a 1-second delay (so the response reaches clients first)
           - "server:status": Return server status (uptime, connections count, config summary)
           - Unknown type: Send "error" with "unknown message type"
         - For request/response messages, echo back the requestId
      5. On 'close': Clean up client metadata, log disconnection
      6. Track connected clients count by role for status reporting

    - The `serverEvents` parameter is an EventEmitter passed in from index.ts. This keeps handler.ts decoupled from server lifecycle (SRP). The handler does NOT import or call server restart functions directly.
    - Return the WebSocketServer instance and a `getClients()` function for inspection

    **Server (`sidecar/src/server.ts`):**
    - Export an async `createServer(config: AppConfig, basePath: string, configStore: ConfigStore, serverEvents: EventEmitter)` function that:
      1. Calls `loadOrGenerateCert(basePath, config)` to get TLS credentials
      2. Creates an Express 5.x app
      3. Adds middleware: express.json(), express.static for public/ directory
      4. Adds a health/status endpoint: `GET /api/status` returns `{ status: "running", version: "0.1.0", uptime: process.uptime() }`
      5. Creates an HTTPS server using Node.js `https.createServer({key, cert}, app)`
      6. Calls `setupWebSocket(server, configStore, serverEvents)` to attach WebSocket
      7. Returns `{ server, app, wss, key, cert }` -- does NOT call listen() yet (caller controls when to start)

    - Export an async `startServer(server, config)` function that:
      1. Calls `server.listen(config.server.port, config.server.host)`
      2. Wraps in a Promise that resolves on 'listening' event
      3. Logs the bound address and port
      4. Calls publishService after server is listening
      5. Returns a `stopServer()` function that: notifies WS clients with close code 1012, closes server, unpublishes mDNS, waits for 'close' event

    - Import ConfigStore, createServer, startServer, and Node.js EventEmitter
    - On startup:
      1. Resolve basePath from process.execPath (use path.dirname)
      2. Override basePath if --config-path arg is provided (for dev mode)
      3. Create `const serverEvents = new EventEmitter()` -- this is the central event bus for server lifecycle
      4. Instantiate ConfigStore with basePath
      5. Log loaded config (redact sensitive fields if any)
      6. Call createServer(config, basePath, configStore, serverEvents) -- passing serverEvents through
      7. Call startServer(server, config)
      8. Log "Server listening on https://{host}:{port}"
    - Keep the stdin-close detection and signal handlers from Plan 01
    - Add graceful shutdown: on SIGTERM/SIGINT, call stopServer(), then process.exit(0)

    IMPORTANT: Use Express 5.x -- it handles async errors with Promises natively. No need for express-async-errors.
    IMPORTANT: Server binds to the specific interface IP from config, NOT 0.0.0.0.
    IMPORTANT: Serve static files from a `public/` directory relative to the sidecar source (for dev) and relative to the binary (for production). Use path.join(basePath, 'public') OR path.join(import.meta.dirname, '../public') depending on whether running compiled or in dev.
