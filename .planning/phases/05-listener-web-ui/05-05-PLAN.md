**Fix E (secondary): Safe pipeline removal in pipeline-manager.ts**

In `pipeline-manager.ts` `removePipeline()` (line 102-115), the issue is `removeAllListeners()` is called immediately after `stop()` resolves. But `stop()` resolves on the child process `exit` event, which can fire BEFORE all stdio streams are fully consumed. Buffered stdout/stderr data arriving after listener removal causes `ERR_UNHANDLED_ERROR` when `GStreamerProcess.emit('error')` fires with zero listeners.

Fix: In `removePipeline()`, after `await pipeline.stop()`, wait for the child process's `close` event (which fires after stdio streams are fully drained) instead of immediately calling `removeAllListeners()`. However, since GStreamerProcess wraps the child and `stop()` already handles the exit event, the simplest fix is:

Add a safety-net no-op error listener BEFORE calling `stop()`, then remove all listeners after a short delay (nextTick or setTimeout(0)) to let stdio drain:

```typescript
async removePipeline(pipelineId: string): Promise<void> {
  const pipeline = this.pipelines.get(pipelineId);
  if (!pipeline) return;

  this.clearRestartTimer(pipelineId);

  // Safety net: catch any errors emitted during shutdown teardown
  // to prevent ERR_UNHANDLED_ERROR when stdio buffers flush after stop
  const safetyErrorHandler = (): void => {};
  pipeline.on("error", safetyErrorHandler);

  await pipeline.stop();

  // Allow one event loop tick for stdio to drain before removing listeners
  await new Promise<void>((resolve) => setImmediate(resolve));
  pipeline.removeAllListeners();

  this.pipelines.delete(pipelineId);
  this.restartAttempts.delete(pipelineId);
  this.restartTimers.delete(pipelineId);

  logger.info(`Pipeline removed: "${pipeline.config.label}"`, { pipelineId });
}
```

**Gap 2 root cause:** (a) OfflineScreen only checks `navigator.onLine`, not server reachability. (b) protoo-client retries forever for previously-connected peers (wasConnected=true triggers infinite `_runWebSocket()` recursion), so the `close` event never fires and `connectionState` never reaches `disconnected`.

**Fix A: Add reconnection timeout in useSignaling.ts**

The protoo-client library has a 10-attempt retry cycle per `_runWebSocket()` call, with each attempt ~2s apart. For previously-connected peers, after exhausting 10 attempts, it calls `_runWebSocket()` again (infinite loop). We cannot change protoo internals, so add an application-level wall-clock timeout.

In `useSignaling.ts`:

1. Add a constant at the top of the file:
```typescript
/** Max time (ms) to stay in "reconnecting" before giving up and showing offline screen. */
const RECONNECT_TIMEOUT_MS = 30_000; // 30 seconds
```

2. Add a `reconnectTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)` to track the timeout.

3. When connectionState transitions to "reconnecting" (in the `disconnected` and `failed` event handlers), start the timer IF it's not already running:
```typescript
// In both "disconnected" and "failed" handlers:
setConnectionState("reconnecting");
if (!reconnectTimerRef.current) {
  reconnectTimerRef.current = setTimeout(() => {
    if (!abortedRef.current) {
      setConnectionState("disconnected");
      // Close the peer to stop protoo's infinite retry loop
      if (peerRef.current && !peerRef.current.closed) {
        peerRef.current.close();
      }
    }
  }, RECONNECT_TIMEOUT_MS);
}
```

4. When connectionState transitions to "connected" (in the `open` handler), clear the timer:
```typescript
// In the "open" handler, after setConnectionState("connected"):
if (reconnectTimerRef.current) {
  clearTimeout(reconnectTimerRef.current);
  reconnectTimerRef.current = null;
}
```

5. In the cleanup function (useEffect return), also clear the timer:
```typescript
if (reconnectTimerRef.current) {
  clearTimeout(reconnectTimerRef.current);
  reconnectTimerRef.current = null;
}
```

**Fix B: Make OfflineScreen respond to connectionState**

Currently OfflineScreen only checks `navigator.onLine`. It needs to also show when `connectionState === "disconnected"` (server unreachable but WiFi is up).

Change OfflineScreen to accept an optional `connectionState` prop:

```typescript
interface OfflineScreenProps {
  connectionState?: ConnectionState;
}

export function OfflineScreen({ connectionState }: OfflineScreenProps) {
  const [isNetworkOffline, setIsNetworkOffline] = useState(!navigator.onLine);
  // ... existing online/offline event listeners unchanged ...

  // Show offline screen when EITHER network is down OR server is unreachable
  const isOffline = isNetworkOffline || connectionState === "disconnected";

  const handleTryAgain = useCallback(() => {
    // If network is actually available, reload the page to re-establish connection
    if (navigator.onLine) {
      window.location.reload();
    }
  }, []);

  if (!isOffline) return null;
  // ... rest of JSX unchanged ...
}
```

Import the `ConnectionState` type from `useSignaling`:
```typescript
import type { ConnectionState } from "../hooks/useSignaling";
```

Update `handleTryAgain`: When the issue is server unreachability (not WiFi down), the old `setIsOffline(false)` would just hide the overlay but the app would still be broken. Instead, use `window.location.reload()` to fully re-establish the signaling connection. This is the correct behavior because once protoo gives up, only a fresh page load can create a new peer.
