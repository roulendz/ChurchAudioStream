**1. Extend ChannelSchema in config/schema.ts:**
Add these optional fields to `ChannelSchema` (after `defaultChannel`):
- `description: z.string().max(200).default("")` — admin-set channel description
- `language: z.object({ code: z.string().default(""), label: z.string().default(""), flag: z.string().default("") }).default(() => ({ code: "", label: "", flag: "" }))` — language with flag emoji, label, and ISO code
- `displayToggles: z.object({ showDescription: z.boolean().default(false), showListenerCount: z.boolean().default(false), showLiveBadge: z.boolean().default(false) }).default(() => ({ showDescription: false, showListenerCount: false, showLiveBadge: false }))` — admin controls what listeners see (minimal defaults per CONTEXT.md locked decision)

**2. Extend ListenerChannelInfo in streaming-types.ts:**
Add these fields to the `ListenerChannelInfo` interface:
- `readonly description: string` — channel description
- `readonly language: { readonly code: string; readonly label: string; readonly flag: string }` — language info
- `readonly listenerCount: number` — current listener count for this channel
- `readonly displayToggles: { readonly showDescription: boolean; readonly showListenerCount: boolean; readonly showLiveBadge: boolean }` — what fields the client should show

**3. Extend ChannelMetadataResolver in router-manager.ts:**
Update the `ChannelMetadataResolver` type to include the new fields (`description`, `language`, `displayToggles`). Update `getActiveChannelList` to populate the new `ListenerChannelInfo` fields from the resolver. Note: `listenerCount` is NOT populated here — it will be injected by SignalingHandler since RouterManager doesn't have access to listener counts.

**4. Update SignalingHandler to enrich channel list with listener counts:**
In `signaling-handler.ts`:
- Create a private method `buildEnrichedChannelList(): ListenerChannelInfo[]` that:
  1. Gets the base list from `routerManager.getActiveChannelList(metadataResolver)`
  2. Enriches each channel with `listenerCount` from `this.getListenerCount(channel.id)` ONLY if `displayToggles.showListenerCount` is true (server optimization per locked decision — don't compute if admin toggled it off)
  3. If `showListenerCount` is false, set `listenerCount: 0` (don't waste CPU iterating peers)
- Update `handlePeer()` to use `buildEnrichedChannelList()` for the initial `activeChannels` notification
- Update `disconnectListenersFromChannel()` to use `buildEnrichedChannelList()` for remaining channels
- Add a new public method `broadcastListenerCounts(): Promise<void>` that:
  1. Calls `buildEnrichedChannelList()` to get channels with current counts
  2. Sends `peer.notify("listenerCounts", { channels })` to all non-admin listeners
  3. This will be called on a 30s interval (the caller — StreamingSubsystem or ListenerWebSocketHandler — sets up the interval)
- Wire the 30s listener count broadcast interval in `ListenerWebSocketHandler` constructor (alongside the existing heartbeat interval). Use the same `heartbeatIntervalMs` config value since it's already 30s default.

**5. Update the metadataResolver creation in streaming-subsystem.ts** (or wherever `buildMetadataResolver` is defined) to include the new fields from config (description, language, displayToggles). Grep for `buildMetadataResolver` to find the exact location.

**Important:** Ensure all existing code that calls `getActiveChannelList` still works — the new fields have sensible defaults (empty string, 0, false).
