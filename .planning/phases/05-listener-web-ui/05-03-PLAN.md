**1. Create `listener/src/lib/connection-quality.ts`:**
Per research code example — WebRTC stats polling:

```typescript
type QualityLevel = "good" | "fair" | "poor";
interface ConnectionQualityResult { level: QualityLevel; rttMs: number; packetLossPercent: number; }
```

- `assessConnectionQuality(consumer: mediasoupTypes.Consumer): Promise<ConnectionQualityResult>` — calls `consumer.getStats()`, extracts RTT from "candidate-pair" report and packet loss from "inbound-rtp" report.
- Thresholds (per discretion recommendation):
  - Good: RTT < 50ms AND packet loss < 1%
  - Fair: RTT < 150ms AND packet loss < 5%
  - Poor: anything worse
- Returns `{ level, rttMs, packetLossPercent }`.
- Handle edge cases: no stats available yet (return "good" as default since we're on local WiFi), missing report fields.

**2. Create `listener/src/components/ConnectionQuality.tsx`:**
Visual indicator component:
- Props: `level: QualityLevel`
- Renders an SVG icon with 3 signal strength bars:
  - Good: all 3 bars filled (green)
  - Fair: 2 bars filled (yellow/amber)
  - Poor: 1 bar filled (red)
- Small size (~24px) positioned in the top-right area of the player
- Tooltip or aria-label: "Connection: good/fair/poor"

**3. Complete `listener/src/views/PlayerView.tsx`:**
Integrate ALL player components into the full-screen player view. This replaces the skeleton from Plan 02:

**Layout (top to bottom):**
- **Header bar:** Back chevron (left), channel name (center), ConnectionQuality icon (right)
- **Channel info area:** Language (flag + label), description (if showDescription enabled), listener count (if showListenerCount enabled, updates every 30s from "listenerCounts" notification)
- **Center: PulsingRing** — animates while playing, stops when muted
- **Elapsed time:** "Listening for 12:34" — timer starts when audio begins playing, resets on channel switch. Use `useState` + `setInterval(1000)` to tick. Format as MM:SS or H:MM:SS.
- **Volume area (lower third):** VolumeSlider with mute toggle icon

**State machine for player states:**
1. `"connecting"` — show "Connecting..." spinner/indicator. Entered when user taps channel card.
2. `"ready"` — show **"Start Listening"** button (large, prominent, centered). Entered when transport + consumer are ready.
3. `"playing"` — show full player UI (ring, volume, elapsed time). Entered after user taps "Start Listening" and AudioContext resumes.
4. `"reconnecting"` — show "Reconnecting..." indicator (per locked decision: auto-reconnect, no manual action). Entered when protoo disconnects while on player screen.
5. `"channel-offline"` — show "Channel offline" message. Stay on player screen (per locked decision). If channel comes back (producerresume or activeChannels update), auto-transition back to "ready" state.
6. `"error"` — show "Can't reach the audio server. Make sure you're on the church WiFi." with retry button. Retry button re-runs the signaling handshake.

**Connection quality polling:**
- Start a 5-second interval that calls `assessConnectionQuality(consumer)` when in "playing" state
- Update the ConnectionQuality component with the result
- Stop polling when leaving player or when consumer is null

**Listener count updates:**
- Subscribe to "listenerCounts" notification from the signaling peer
- Update the per-channel listener count displayed on the player screen
- Same 30s interval as the channel list (server pushes it)

**Auto-reconnect behavior (per locked decisions):**
- When protoo peer fires "open" after disconnection while on player screen: auto-reconnect to the same channel WITHOUT requiring "Start Listening" tap (exception per locked decision: session already had user gesture)
- When consumer fires "producerclose": transition to "channel-offline" state
- When "activeChannels" notification shows the current channel is back: auto-reconnect

**Back navigation:**
- Left-aligned chevron in header (per discretion recommendation)
- No separate stop button (back = stop per discretion recommendation)
- On tap: disconnect mediasoup transport (audio stops immediately), transition to channel list
