# Requirements: ChurchAudioStream

**Defined:** 2026-02-05
**Core Value:** Enable multilingual church members and hearing-impaired listeners to hear sermons in their language through their own phones, using the church's existing audio infrastructure -- with near-zero latency and zero friction.

## v1 Requirements

### Audio Capture

- [ ] **CAPT-01**: App can receive AES67 multicast RTP streams from a Dante network
- [ ] **CAPT-02**: App can capture audio from local system audio devices (USB interfaces, microphones, line-in, virtual devices)
- [ ] **CAPT-03**: Admin can configure each channel to use either Dante/AES67 network source or local audio device
- [ ] **CAPT-04**: App discovers available AES67 streams on the network automatically
- [ ] **CAPT-05**: App lists available system audio input devices for selection

### Audio Processing

- [ ] **PROC-01**: Audio normalization/AGC applied per channel to even out volume levels
- [ ] **PROC-02**: Speech/Music mode toggle per channel to prevent audio processing from destroying non-speech content (choir, instruments)
- [ ] **PROC-03**: GStreamer encodes processed audio to Opus at configurable bitrate (~120kbps default)
- [ ] **PROC-04**: Each channel runs as an independent GStreamer process for fault isolation

### WebRTC Streaming

- [ ] **STRM-01**: mediasoup SFU distributes audio streams to multiple simultaneous browser listeners
- [ ] **STRM-02**: Automatic reconnection on WiFi drops with exponential backoff and ICE restart
- [ ] **STRM-03**: Listener can blend two channels (original + translation) with a mix balance slider
- [ ] **STRM-04**: Listener can toggle audio processing (normalization) on/off from their phone
- [ ] **STRM-05**: Sub-100ms end-to-end latency from Dante input to listener's earbuds

### Listener Web UI

- [ ] **LWEB-01**: Welcome screen with channel selection (big tap targets, mobile-first)
- [ ] **LWEB-02**: Per-stream volume control slider
- [ ] **LWEB-03**: Channel switching without returning to home screen
- [ ] **LWEB-04**: PWA support: Add to Home Screen, cached assets, remembered preferences
- [ ] **LWEB-05**: QR code generated by admin for easy phone access to Web UI
- [ ] **LWEB-06**: Web UI available in multiple languages (localization)
- [ ] **LWEB-07**: Light/dark theme with system-adaptive auto-detection and manual override

### Admin Desktop GUI

- [ ] **AGUI-01**: Dashboard layout with sidebar navigation
- [ ] **AGUI-02**: Channel configuration: display name, input source (Dante or device), visibility, ordering
- [ ] **AGUI-03**: Per-channel audio processing controls (normalization level, Speech/Music mode)
- [ ] **AGUI-04**: Real-time VU meters showing audio levels per channel
- [ ] **AGUI-05**: Listener count per channel and total
- [ ] **AGUI-06**: Server status display (CPU, memory, active connections)
- [ ] **AGUI-07**: Stream health monitoring (latency, packet loss, buffer status per channel)
- [ ] **AGUI-08**: Statistics and graphs for engagement (listener trends, peak usage)
- [ ] **AGUI-09**: Web server port configuration
- [ ] **AGUI-10**: QR code display/generation for listener access
- [ ] **AGUI-11**: Light/dark theme with system-adaptive auto-detection and manual override

### Configuration & Persistence

- [ ] **CONF-01**: All settings stored in JSON config file
- [ ] **CONF-02**: Settings persist across app restarts
- [ ] **CONF-03**: Import/export settings as JSON file
- [ ] **CONF-04**: Web server runs on configurable IP:port

### Platform & Operations

- [ ] **PLAT-01**: Cross-platform Tauri 2.x builds for Windows, Mac, and Linux with Node.js sidecar for server logic (mediasoup, Express, WebSocket)
- [ ] **PLAT-02**: Installer packages (.exe, .dmg, .deb) for permanent installation
- [ ] **PLAT-03**: Portable app option (no install needed)
- [ ] **PLAT-04**: Optional auto-start on boot (configurable in settings)
- [ ] **PLAT-05**: Update notification system (check for new versions, notify admin)

### Reliability & Self-Healing

- [ ] **RELY-01**: Auto-restart crashed GStreamer pipelines without affecting other channels
- [ ] **RELY-02**: mediasoup worker health monitoring and auto-rotation for long sessions
- [ ] **RELY-03**: Network diagnostic tool (IGMP snooping check, PTP status, multicast health)
- [ ] **RELY-04**: Graceful degradation when individual channels fail
- [ ] **RELY-05**: Connection health indicators visible in both admin GUI and listener Web UI

## v2 Requirements

### Recording & Archiving

- **RECD-01**: Record individual channels to audio files during service
- **RECD-02**: Playback of recorded sessions from Web UI

### Remote Streaming

- **REMT-01**: Optional TURN server configuration for internet listeners
- **REMT-02**: Bandwidth management for remote connections

### Enhanced Features

- **ENHC-01**: AI noise cancellation (RNNoise/DTLN) per channel
- **ENHC-02**: Per-channel EQ (3-5 band equalizer)
- **ENHC-03**: Church branding config (logo, colors, welcome message)
- **ENHC-04**: Accessibility features (screen reader support, high contrast mode)
- **ENHC-05**: Admin GUI localization (beyond English)
- **ENHC-06**: Auracast Bluetooth output (when standard matures)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Captive portal (auto-redirect on WiFi) | Fragile across devices, breaks HTTPS, requires router-level config. Use QR codes instead. |
| Native mobile apps (iOS/Android) | PWA covers 90% of value without app store friction. |
| AI-powered automatic translation | Unreliable for religious content, 2-5s latency, requires cloud. Human interpreters are better. |
| Video streaming | Different technical domain, would saturate church WiFi. Stay audio-only. |
| Chat/messaging during service | Scope creep, moderation burden, distraction from worship. |
| Listener authentication/accounts | Friction kills adoption. Churches are welcoming. Anonymous listener counts suffice. |
| Hearing loop (T-coil) integration | Purely hardware (electromagnetic). Users can pair hearing aids to phone via Bluetooth. |
| Multi-site/campus support | v2+ scope -- requires fundamentally different architecture. |
