# Plan: Multi-Source Audio Mixing Refactor (260429-hb3)

## Goal

Replace per-source gst-launch processes with **one gst-launch process per channel** that combines multiple sources via GStreamer `audiomixer`. Eliminates the bind-port + SSRC race that today kills all but one source per channel. Sidecar-only refactor; no listener/admin/Tauri changes.

Mental model: `channelPipelines: Map<channelId, pipelineId>`. channelId stable, pipelineId rotates on every replace. SSRC + sender bind-port derived from channelId → comedia tuple stable across restarts → no producer-replacement churn.

## Risks Captured (from research)

1. **Audiomixer `latency=10000000` (10ms) consumes 10ms of the 100ms total budget** (RESEARCH §1). Sub-budget already includes WASAPI buffer-time=20ms, audioloudnorm 192kHz roundtrip ~2-5ms, Opus frame-size=20ms, network jitter ~20ms. Margin tight (~25ms). Watch UAT for audible delay; tune to 5ms only if `output-buffer-duration` adjusted accordingly.
2. **Mixer + AGC chain caps roundtrip not empirically tested** (RESEARCH §6 Q4). Smoke test was mixer→fakesink only. The full `audiomixer ! audio/x-raw,rate=48000,channels=2 ! audioconvert ! audioresample ! audio/x-raw,rate=192000 ! audioloudnorm ! ...` chain may surface caps negotiation issues. UAT is the gate. If "not-negotiated" fires, suspect missing intermediate `audioconvert` between mixer output and AGC input.
3. **`alignment-threshold=40000000` (40ms default)** + clock drift between WASAPI and AES67 sources may emit `discont` warnings. RESEARCH §6 Q3 says monitor stderr; tune if observed. Existing `attachStderrErrorDetector` will surface as `pipeline-error` event — check log for `discont` substring before declaring crash.
4. **EOS handling for file-only channels** (RESEARCH §6) — file-loop logic moved to channel-manager. Loop trigger relies on `wasStopRequested === false && code === 0 && shouldLoopOnEos === true`. Edge case: mixed file + live channel where the file EOSes but live keeps streaming — audiomixer with `force-live=false` (default) only EOSes when ALL inputs are done, so the live source keeps the channel up. ✓ Correct, per RESEARCH §6.
5. **`replacePipeline` 400ms socket-release window**: existing `WINDOWS_SOCKET_RELEASE_DELAY_MS` already covers this (RESEARCH §3). Do NOT add additional delays in `replacePipeline` — that would double-count and add user-visible latency on every source change.
6. **comedia tuple stability**: SSRC + bind-port deterministic per channelId. Replace cycle preserves tuple → mediasoup never re-locks → `bytesReceived` monotonic. Verified by debug log "follow-up #2" (RESEARCH §3). If a refactor regression breaks SSRC/port determinism, comedia silently drops packets — listener UI will show channel offline. Treat any post-refactor "channel offline" UAT report as a port/SSRC regression.
