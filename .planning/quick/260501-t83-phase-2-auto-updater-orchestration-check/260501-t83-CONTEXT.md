<decisions>
## Implementation Decisions

User skipped re-discussion (plan §10: "This plan IS the discuss phase. When stuck, run /gsd-debug not /gsd-discuss-phase.").
Phase 1 set the precedent: lock-down decisions in CONTEXT.md, proceed without live questioning.

### `dispatcher.rs` scope
- **Decision:** Phase 2 dispatcher is **payload-type definitions only** — no `app.emit()`, no Tauri imports, no async functions.
- **Why:** User explicitly said "No Tauri plugin yet" + plan §3 Phase 3 step 6 owns the `app.emit("update:available", ...)` calls. Phase 2 stays pure-Rust testable. Defining payload structs early lets Phase 3 wire them in without redesign.
- **Concrete contents:** `UpdateAvailablePayload { version, notes, download_url }`, `UpdateDownloadProgressPayload { downloaded_bytes, total_bytes }`, `UpdateInstalledPayload { version }`. All `#[derive(Serialize, Clone, Debug)]`. No `Deserialize` (these are emit-only outbound types).
- **Tests:** 1-2 round-trip-via-serde-json tests to lock the wire format.

### `UpdateDecision` enum design
- **Decision:** `pub enum UpdateDecision { Notify { version: String, notes: String, platform_url: String }, SilentSkip(String), NoUpdate }`.
- **Why:** Plan §3 Phase 2 evaluate_update sketch returns `UpdateDecision::Notify` (no fields shown), but caller in Phase 3 needs the version + notes + platform-specific URL to feed into `UpdateAvailablePayload`. Carrying these fields inside `Notify` keeps `evaluate_update` SRP-clean — caller doesn't reach back into the manifest.
- **`SilentSkip(String)`:** the reason string. Plan uses `"user skipped"`, `"dismissed cooldown"`, `"bad manifest: {e}"`, `"bad current: {e}"` exactly. Keep as `String` for now; if a Phase 3 caller needs to branch on reason, refactor to enum.
- **`NoUpdate`:** unit variant.

### Storage file format
- **Decision:** JSON via `serde_json` (already in `Cargo.toml`).
- **Why:** Human-readable, no new deps. File is small (3 fields), tiny IO budget, no perf concern. Tauri ecosystem convention.
- **Path:** caller passes a `&Path`. Plan §3 says `$APPDATA/com.churchaudiostream.app/update-state.json` but that path resolution lives in Phase 3 (uses `app_handle.path().app_data_dir()`). Phase 2 stays pure: take a path, read/write that path.
- **`load()` semantics:** if the file does not exist OR is empty, return `Ok(UpdateState::default())`. This matches plan test `storage_load_returns_default_when_missing()`. Other IO/parse errors propagate.

### Pure mutation helpers contract
- **Decision:** all three helpers (`with_dismissed_now`, `with_skipped_version`, `with_check_completed`) take `state: UpdateState` by VALUE and return `UpdateState`. No `&mut self`. Plan calls these "pure mutation helpers (return new state, don't mutate in place)" explicitly.
- **`with_skipped_version` dedup:** if version already in `skipped_versions`, return state unchanged (no-op). Plan test: `with_skipped_version_dedupes`.
- **`with_dismissed_now`:** sets `last_dismissed_unix = now_unix`, leaves rest. Plan test: `with_dismissed_now_does_not_mutate_input` (proves input is moved, output is new).

### Error types
- **Decision:** new `StorageError` enum in `storage.rs` (hand-rolled, manual `Display` + `std::error::Error`, matches Phase 1 pattern). Variants: `Io(std::io::Error)`, `Parse(serde_json::Error)`. No `thiserror`.
- **Decision:** `checker.rs` does NOT need its own error type. `evaluate_update` returns `UpdateDecision` (with reason strings inside `SilentSkip`). `should_check_now` returns `bool`. `is_version_skipped` returns `bool`. None can fail.

### Tiger-Style assertions
- `should_check_now`: `debug_assert!(now_unix >= last_check_unix)` (plan §2 Layer 2 mandates this).
- `evaluate_update`: flat structure with early returns, no nested ifs (plan shows the GOOD vs BAD example).
- All public functions ≤ 50 lines.

### `is_version_skipped` signature
- **Decision:** `pub fn is_version_skipped(version: &str, skipped: &[String]) -> bool` per plan signature.
- **Implementation:** `skipped.iter().any(|v| v == version)`. One-liner.

### Tests strategy
- **Decision:** inline `#[cfg(test)] mod tests` per file (matches Phase 1).
- **Storage round-trip test:** use `tempfile::tempdir()` for filesystem isolation. Add `tempfile = "3"` as `[dev-dependencies]` (NOT runtime dep — only for tests).
- **Plan-listed tests (12 total minimum):**
  - `should_check_when_never_checked`
  - `should_not_check_within_cooldown`
  - `should_check_after_cooldown_expires`
  - `evaluate_returns_notify_for_new_version`
  - `evaluate_returns_silent_skip_when_skipped`
  - `evaluate_returns_silent_skip_within_dismiss_cooldown`
  - `evaluate_returns_no_update_when_current_latest`
  - `evaluate_returns_no_update_when_downgrade`
  - `with_dismissed_now_does_not_mutate_input`
  - `with_skipped_version_dedupes`
  - `storage_round_trip`
  - `storage_load_returns_default_when_missing`
- **Plus dispatcher payload tests (~2):** `update_available_payload_serializes`, `update_download_progress_payload_serializes`.
- Target: ≥14 new tests, total ≥36 across `update::*` after Phase 2.

### What we will NOT add to `Cargo.toml`
- ❌ `thiserror` / `anyhow` — hand-rolled errors continue.
- ❌ `tauri-plugin-updater` — Phase 3.
- ❌ `chrono` / `time` — Unix epoch i64 is sufficient.
- ❌ `tracing` / `log` — no logging in pure functions.
- ✅ `tempfile = "3"` under `[dev-dependencies]` only — for `storage_round_trip` test isolation.
- (`serde` + `serde_json` already present.)

### Cross-cutting from Phase 1 review (defer notes)
- Phase 1 review flagged URL strict-prefix laxness, HashMap non-determinism, etc. Those live in `manifest.rs` and are NOT in scope for Phase 2. Do NOT touch `manifest.rs` or `version.rs` — Phase 2 only ADDS modules.

### Claude's Discretion (areas not specified by plan)
- `UpdateState::default()` derivation: `#[derive(Default)]` on the struct (all fields default to `0` / `Vec::new()`).
- File-not-found detection in `load()`: use `std::io::ErrorKind::NotFound` — clean and platform-agnostic.
- Empty-file detection: read content first, check `.is_empty()`, return default. Avoids serde_json error on empty input.
- Module ordering in `mod.rs` after Phase 2: `pub mod checker; pub mod dispatcher; pub mod manifest; pub mod storage; pub mod version;` (alphabetical).
- Selected re-exports from `mod.rs`: `pub use checker::{UpdateDecision, evaluate_update, should_check_now, is_version_skipped}; pub use storage::{UpdateState, load, save, with_dismissed_now, with_skipped_version, with_check_completed, StorageError};` etc. — only what Phase 3 will call.

## Specific Ideas

- Plan signatures (plan §3 Phase 2) are the contract. Don't drift from them; Phase 3 depends on these names.
- The `evaluate_update` "GOOD" example in plan §3 Phase 2 (flat early-returns) is the implementation template. Copy its structure, don't reinvent.
